# OpenSIPS Constitution

## 核心原则

### I. 内存安全与代码质量

OpenSIPS 核心和所有模块必须遵循内存安全原则：禁止缓冲区溢出、使用安全的字符串函数、正确管理内存生命周期。所有新增代码必须通过静态分析工具（Coverity Scan、cppcheck）和编译器警告（-Wall -Werror）检查。

**理由：** 作为电信级 SIP 服务器，内存错误可能导致崩溃、安全漏洞或通话中断，这与项目的高可用性目标不可接受。

### II. SIP 协议合规性

所有 SIP 协议实现必须严格遵循 RFC 3261 及相关扩展（RFC 3262、RFC 3515、RFC 4028 等）。偏离标准必须有明确文档说明和充分的技术理由。

**理由：** 互操作性是 SIP 服务器的核心价值，只有遵循标准才能与其他 SIP 设备和服务正常通信。

### III. 模块化架构

OpenSIPS 采用模块化设计：核心（core）提供基础设施，模块（modules）提供独立功能。模块必须具有清晰的接口定义、独立的单元测试，且不得直接访问其他模块的内部状态（除非通过正式导出的 API）。

**理由：** 模块化确保系统可维护、可测试和可扩展，允许用户按需加载功能。

### IV. 性能与资源效率

所有代码必须考虑性能影响。对于关键路径（消息处理、事务创建、对话框管理），必须提供算法复杂度说明。禁止在快速路径中引入非必要的锁、日志输出或内存分配。

**理由：** SIP 服务器需要处理大量并发连接，性能瓶颈会直接影响通话质量和用户体验。

### V. 安全第一

安全不是事后考虑：所有涉及认证、授权、加密的功能必须从头设计。新增安全相关代码必须经过代码审查。敏感数据（密码、密钥）必须加密存储或使用后立即清除。

**理由：** SIP 服务器是网络基础设施，安全性漏洞可能被利用进行窃听、欺诈或服务中断。

## 附加约束

### 技术栈要求

- **语言**：C99 标准，优先使用现代 C 特性
- **构建系统**：Makefile + GCC/Clang
- **测试**：单元测试框架（BUnit/Bessel 风格）、集成测试、模糊测试（OSS-Fuzz）
- **CI/CD**：GitHub Actions，必须通过所有测试才能合并
- **静态分析**：Coverity Scan、cppcheck、scan-build
- **平台**：Linux（主要）、macOS（开发）、多架构支持（x86_64、ARM64）

### 合规标准

- GPL v2 许可（所有代码）
- SIP RFC 合规性
- TLS/加密遵循 TLS 1.2+ 标准

## 开发流程

### 代码审查要求

- 所有 PR 必须经过至少一名维护者审查
- 重大变更（核心修改、API 变更、性能影响）需要两名审查者
- 审查者必须检查：内存安全、协议合规性、性能影响、安全性
- CI 必须全部通过（单元测试、集成测试、静态分析）

### 质量门禁

- 单元测试覆盖率：核心模块不低于 80%
- 静态分析：零严重警告（高优先级）
- 模糊测试：新增模糊目标必须通过 OSS-Fuzz 验证
- 性能回归：禁止引入超过 5% 的性能下降（除非有明确优化）

### 发布流程

- 版本号遵循语义化版本（MAJOR.MINOR.PATCH）
- 每次发布必须更新 CHANGELOG
- 安全修复作为 PATCH 版本立即发布

## 治理

### 章程优先级

本章程优先于所有其他开发实践和文档。若章程与具体实现指南冲突，以本章程为准。

### 修订程序

1. 修订案必须包含：变更理由、影响分析、迁移计划（若涉及）
2. 修订需要 Pull Request 形式，经过至少两名维护者批准
3. 重大修订（移除或重新定义核心原则）需要公开讨论（devel@lists.opensips.org）
4. 版本号根据变更类型递增：MAJOR（破坏性变更）、MINOR（新增/扩展）、PATCH（澄清/修正）

### 合规验证

- 所有 PR 必须通过自动化合规检查
- 维护者在合并前必须验证章程遵循
- 复杂度增加必须有充分理由，并在 plan.md 中记录

### 参考文档

- 运行时开发指导：参见项目根目录 `CREDITS`、`AUTHORS`、`COPYING`
- 模块开发指南：各模块目录下的 `README`
